# ReturnOrder Boundary Design

This document locks the `ReturnOrder` boundary before any schema, migration, route, DTO, WebAdmin UI, mobile/member/storefront contract, payment/refund flow, shipment flow, invoice archive/download behavior, or production flow changes.

The v1 policy is explicit: refund and restock are allowed only after goods are received and inspected. A request or approval alone never makes refund or stock movement eligible.

## Current Darwin Return/Refund/Shipment Findings

- The current `Refund` model is the authoritative payment/order/invoice settlement record. It is not a `ReturnOrder` and must not be stretched into return request, receipt, inspection, or stock disposition.
- `QueueDhlReturnShipmentHandler` supports return shipment/label orchestration, but it does not model received quantity, inspection result, accepted/rejected return quantity, restock, scrap, or damaged-goods disposition.
- `ShipmentLine.Quantity` and `DeliveryNoteLine` represent shipped/documented quantity. Darwin does not yet have an official return document for requested, approved, received, accepted, rejected, scrapped, or restocked quantities.
- Inventory reservation/allocation exists in order status flows, but return receipt, restock, scrap, and damaged-goods policy is not yet modeled as a formal return lifecycle.
- `CreditNote` remains finance-gated. Darwin must not create a negative invoice or non-posting credit-note document before accounting, tax, posting, numbering, archive, and settlement policy are designed.

## Locked ReturnOrder Decisions

- `ReturnOrder` is a formal internal/WebAdmin Sales document. It does not replace `Refund`, `Shipment`, `DeliveryNote`, `Invoice`, or future `CreditNote`.
- The v1 lifecycle is `Requested`, `Approved`, `Rejected`, `ReturnShipmentQueued`, `Received`, `Inspected`, `RefundReady`, `Refunded`, `Closed`, and `Cancelled`.
- Requested quantity comes from an operator/customer return request. Approved quantity must not exceed shipped/documented quantity.
- Received quantity is recorded only when goods are physically received.
- Accepted, rejected, scrapped, and restocked quantities are recorded only after inspection.
- Refund eligibility starts only after inspection and only for accepted quantities. Current refund/payment handlers remain the mutation owner for actual refund settlement.
- Restock is allowed only after inspection. Non-catalog order lines can be returned for service/commercial evidence, but they do not create stock movement. Damaged or scrapped quantities must not return to sellable stock.
- `ReturnOrder` numbering uses `NumberSequenceDocumentType.ReturnOrder` at approval or the first official state. Requested drafts can remain internal without an official number.
- Customer evidence, photos, files, and supporting documents use `DocumentRecord`. Lifecycle evidence uses deterministic `BusinessEvent/AuditTrail`. External RMA/import identities use `ExternalReference`.
- Public WebApi, mobile/member, storefront checkout, payment-intent, invoice archive/download, structured JSON/XML, and issued snapshot contracts remain unchanged in v1.
- `CreditNote` stays blocked until Finance/accounting design defines posting, tax evidence, legal numbering, archive/source-model behavior, and settlement alignment.

## ReturnOrder Core Model Outcome

Darwin now implements `ReturnOrder`, `ReturnOrderLine`, and `ReturnOrderRefundLink` as additive Sales schema entities for internal/WebAdmin return lifecycle control.

- `ReturnOrder` is a formal Sales document. It does not replace `Refund`, `Shipment`, `DeliveryNote`, `Invoice`, or future `CreditNote`.
- The implemented lifecycle follows the locked boundary: `Requested`, `Approved`, `Rejected`, `ReturnShipmentQueued`, `Received`, `Inspected`, `RefundReady`, `Refunded`, `Closed`, and `Cancelled`.
- Requested, approved, received, accepted, rejected, scrapped, and restock quantities are stored separately on return lines. Refund and restock are rejected before inspection.
- Approval reserves `NumberSequenceDocumentType.ReturnOrder` exactly once. Requested returns can exist without an official return number.
- Line snapshots come from current `OrderLine` and optional `ShipmentLine` records. Catalog and non-catalog lines are both supported without fake products or live catalog recomputation.
- Restock uses the existing inventory return receipt handler with `ReferenceId = ReturnOrderId`. Only variant-backed accepted/restock quantities can create stock movement.
- Refund settlement remains owned by current refund/payment handlers. `ReturnOrderRefundLink` links completed refunds back to the return for reconciliation and evidence.
- Lifecycle evidence is written through deterministic `BusinessEvent/AuditTrail` records with safe/reportable payloads only.
- WebAdmin Sales now includes Return Orders navigation, list/filtering, create request, detail, approve/reject, queue return shipment, receive, inspect, mark refund ready, link refund, mark refunded, close, and cancel actions.
- Public WebApi, mobile/member, storefront checkout, payment-intent routes, invoice archive/download, structured source-model, and issued snapshot contracts remain unchanged.
- `CreditNote` remains finance-gated. No negative invoice or non-posting credit-note document was introduced.

## ReturnOrder Refund And Inventory Reconciliation Hardening Outcome

Darwin now hardens ReturnOrder reconciliation without adding a new domain concept, schema, public API, mobile/member exposure, storefront flow, credit note, or parallel ledger.

- `LinkRefund` accepts only completed, active refunds from the same order and currency. Pending, failed, deleted, other-order, and other-currency refunds are rejected.
- Duplicate refund links are idempotent and do not create a second link or double-count the refund amount.
- Multiple completed refunds can reconcile one return. A partial linked total keeps the return in `RefundReady`; the return moves to `Refunded` only when linked refunds exactly cover `RefundEligibleGrossMinor`.
- Linked refund totals cannot exceed the return's eligible gross amount.
- Inspection/restock remains after-receipt and after-inspection only. Retry behavior is guarded so the inventory return receipt does not create duplicate stock movement for the same `ReturnOrderId`.
- Non-catalog lines can carry return, inspection, and refund evidence, but restock is rejected because there is no product variant to move in inventory.
- WebAdmin Return Orders now show linked and remaining refund amounts in list/detail views. Quantity inputs remain limited to create, approve, receive, and inspect stages; refund-ready linkage accepts only refund id evidence.
- Current `Refund` remains the authoritative settlement record, and the existing inventory return receipt owner remains the authoritative stock movement path.
- `CreditNote` remains finance-gated. No negative invoice or non-posting credit-note document was introduced.
