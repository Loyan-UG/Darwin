# Foundation Implementation Consolidation Review

This review consolidates the release-sensitive foundation slices completed before broader ERP domain expansion continues. It is a decision and evidence artifact only. It does not change entities, migrations, API routes, DTOs, mobile contracts, or production code.

If any real mobile, loyalty, or foundation gap is discovered after this review, it must be completed before mobile release with matching WebApi, contract, mobile shared service, and test updates. Release delay is acceptable when the alternative is post-release downtime, duplicated source-of-truth state, or forced mobile migration.

## Current P0 Foundation Baseline

- Address foundation: an internal canonical mapper exists and is guarded across `Address`, `CustomerAddress`, `BusinessLocation`, public business location mapping, and checkout order snapshots.
- Mobile API boundary: route and DTO freeze rules are documented, and route/service/serialization guards exist for release-sensitive mobile surfaces.
- Loyalty ledger and scan: loyalty ledger remains the points source of truth, and QR/session contracts remain opaque-token-first without internal identifier leakage.
- Identity profile and customer bridge: `User` is the source for linked member identity/profile/contact; `Customer` keeps CRM lifecycle, segmentation, tax/commercial, and fallback data.
- Business and BusinessMember access: `Business` remains the merchant root, and `BusinessMember` remains the current source of business mobile access.
- Order and invoice snapshots: member order/invoice snapshots, archive/download routes, and structured source-model export routes are guarded and remain part of the current order/invoice foundation.
- External-system readiness: `ExternalSystem`, `ExternalReference`, and `SourceOfTruth` exist as structured foundation primitives for external identity and coexistence.
- Customization and collaboration foundation: `CustomFieldDefinition`, `CustomFieldValue`, `Activity`, `Note`, and `DocumentRecord` exist for low-frequency custom data, shared timelines, human notes, and document metadata.
- Number sequencing: `NumberSequence` and the internal sequence service exist for future ERP document numbers while preserving existing issued numbers and current mobile contracts.
- Event and audit foundation: `BusinessEvent` and `AuditTrail` exist as shared evidence primitives without replacing specialized ledgers, archives, provider records, or operational audits.
- Module visibility: `FeatureArea` and `BusinessFeatureOverride` exist for logical module visibility and packaging without replacing permissions, roles, launch guards, or access-state.

## Consolidated Decisions

- Release-sensitive hardening for the main mobile/loyalty/foundation surfaces is in place.
- The shared foundation primitives needed before broad ERP expansion have been implemented at the internal domain/application/persistence level.
- `SyncState` and `SyncConflict` remain deferred unless a concrete integration flow proves they are needed.
- No future ERP module may create a parallel source of truth for loyalty balance, business access, linked member profile/contact, order snapshots, issued invoice source documents, external identity, custom fields, document metadata, document numbering, audit evidence, or module visibility.
- Broad module expansion has moved through CRM design and the first CRM core model expansion; CRM should next integrate foundation primitives only where concrete flows need them.

## Next Implementation Slice

`CRM Foundation Primitive Integration Slice`

Goal: connect CRM flows to shared foundation primitives where they provide real value, without duplicating source-of-truth state or changing mobile/member contracts.

Default first scope:

- Use `ExternalReference` for imported or synchronized CRM identities.
- Add `Activity` and `Note` projections only where CRM needs shared timeline behavior beyond existing `Interaction`.
- Register CRM attachment metadata with `DocumentRecord` without changing upload/download flows.
- Emit `BusinessEvent` or `AuditTrail` only for important lifecycle transitions that need evidence.
- Keep loyalty/mobile contracts unchanged unless a release-sensitive gap is found.

Broad CRM UI/API expansion should wait until these primitive usage points are explicit.
