# Foundation Primitives Design

This document turns the release-sensitive foundation review into implementation-ready design decisions. It does not change entities, migrations, APIs, mobile contracts, or tests. Any implementation work must happen in a later step after these decisions are reviewed.

The design uses Darwin-owned English names only. It records current Darwin model findings and target ERP scope decisions without importing external schemas, vendor naming, deployment-specific customizations, or non-English names.

## Current Darwin Model Findings

- `Address`, `CustomerAddress`, `BusinessLocation`, and order/invoice snapshots currently represent address-like data in multiple shapes. This is workable short term, but a canonical address decision is required before release-sensitive mobile contracts and later ERP sales, purchasing, shipping, warehouse, and finance records depend on divergent mappings.
- `User` currently combines identity, profile, preferences, consent-like fields, attribution, and CRM-adjacent fields. It directly affects mobile auth, profile, phone verification, external login, token refresh, push registration, and account deletion flows.
- `Business` is already relatively complete for mobile/business operations, public discovery, branding, communication preferences, subscription status, and onboarding. `BusinessMember` is intentionally lightweight today, but its future linkage to staff/employee records must be decided before mobile release to avoid incompatible access-state and role concepts.
- `Customer` already has a `UserId` bridge, while fallback profile and address data still exist for guest, lead, and CRM flows. The source-of-truth rule must be explicit: linked customers resolve profile/contact data from identity; unlinked customers retain CRM-owned fallback data.
- `LoyaltyAccount`, scan sessions, QR/session records, loyalty contracts, and mobile routes are release-sensitive. The loyalty ledger remains the points source of truth, but required contract, ledger, identity-link, or reporting changes must be implemented before mobile release.
- `Order` and `Invoice` intentionally keep immutable snapshots. Later sales and finance expansion must evolve these models and their projections instead of creating a parallel order or invoice model.
- `ApiRoutes`, mobile shared services, launch readiness guards, and contract serialization tests form the current mobile API boundary. Any breaking route or DTO change must be made deliberately before mobile release, with compatibility tests updated in the same implementation step.

## Foundation Primitive Decisions

| Primitive | Purpose | Current overlap in Darwin | Recommended shape | Applies to first | Mobile/Loyalty release impact | Required before mobile release | Implementation phase | Notes |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `ExternalSystem` | Register external business systems and trust boundaries for import, export, sync, and coexistence. | Provider callbacks, webhook delivery, shipment provider operations, analytics exports, and site settings cover specific integration cases but not a canonical external-system registry. | Structured entity with real columns for name, system type, status, owner scope, trust boundary, default source-of-truth policy, and operational flags. | Business, Customer, Order, Invoice, Payment, Shipment, Supplier later. | Possible if mobile-visible records are externally sourced or synchronized. | No for first mobile release unless a mobile-visible record needs external source labeling. | Phase 2 foundation implementation. | Do not store secrets in this entity; credentials remain in secure configuration or provider-specific protected storage. |
| `ExternalReference` | Attach stable external identifiers to Darwin-owned records without adding provider-specific columns to every entity. | `User.ExternalIdsJson`, payment provider references, shipment provider references, invoice archive metadata, and provider callback inbox records solve local cases. | Structured child entity or owned collection with real columns for entity type, entity id, external system id, external id, reference kind, status, and uniqueness rules. | Customer, Business, Order, Invoice, Payment, Shipment; User only if release review proves it is required. | High if added to `User`, `Business`, `Customer`, loyalty, order, invoice, or mobile-facing projections. | Yes only for release-sensitive records that require integration identity before launch; otherwise defer. | Phase 1 only for required release-sensitive records; Phase 2 for broad rollout. | Prefer structured references over JSON when filtering, reconciliation, sync, support lookup, or conflict handling matters. |
| `SourceOfTruth` | Record which system or aggregate currently owns a field group or record lifecycle. | Some comments and handler rules imply ownership, for example identity-owned profile data for linked customers and loyalty-owned points balances. | Enum/value object on sync-aware records plus documented ownership rules; use real columns where ownership affects behavior, filtering, or conflict resolution. | Customer/User bridge, loyalty points ownership, order/invoice snapshots, externally sourced records. | High for profile, customer, loyalty balance, and invoice/order projections. | Yes for `User`/`Customer` profile ownership and loyalty balance ownership; broad sync ownership can defer. | Phase 1 design decision, Phase 2 broad implementation. | Default: linked customer profile/contact data comes from `User`; loyalty balances come only from loyalty ledger; issued snapshots remain immutable historical evidence. |
| `SyncState` | Track inbound/outbound sync status, timestamps, versions, and retry state. | Provider callback inbox, webhook delivery, dispatch operations, shipment provider operations, and analytics export jobs have operation-specific status fields. | Structured sync state columns or side table for records that participate in repeatable import/export; real columns for status, direction, last synced, version/checksum, and retry marker. | External references for Customer, Business, Order, Invoice, Payment, Shipment; inventory and finance later. | Medium if mobile edits can conflict with external sync or if mobile displays synced data. | No for first mobile release unless release-sensitive records are actively synchronized. | Phase 2 foundation implementation. | Keep raw provider payloads in explicit payload/snapshot records, not in general sync state columns. |
| `SyncConflict` | Capture conflicting changes that need operator or automated resolution. | No canonical conflict entity found; current operational queues handle provider callbacks and shipping/payment exceptions separately. | Structured entity with real columns for entity type, entity id, external system id, field group, local value snapshot, remote value snapshot, status, resolution, and audit fields. | Customer, Business, Order, Invoice, Payment, Shipment; inventory and finance later. | Medium if mobile-visible records can be edited both internally and externally. | No unless a release-sensitive external sync is enabled before mobile launch. | Phase 2 foundation implementation. | Do not expose raw conflict payloads to mobile clients; mobile may receive stable status/projection only when needed. |
| `CustomFieldDefinition` | Define customer-specific, optional, uncertain, or vertical-specific fields without adding canonical columns. | Several JSON fields exist for preferences, rules, opening hours, admin text overrides, attribution, and provider/source payloads. | Structured definition entity with target entity type, key, label, data type, validation hints, visibility, required flag, active flag, and permission scope. | Customer, Business, Supplier, Product, Order, Invoice later; not first-choice for mobile profile fields. | Low unless exposed in mobile forms or mobile filtering. | No for first mobile release. | Phase 2 foundation implementation. | Do not use custom fields for common, reportable, compliance-relevant, integration-key, or cross-module data. |
| `CustomFieldValue` | Store values for approved custom field definitions. | Current ad hoc JSON fields carry some extensibility payloads, but without shared definitions or typed querying. | Structured typed value rows for reportable custom fields; JSON value only for low-frequency or unstructured data. | Same targets as `CustomFieldDefinition`. | Low unless mobile reads or writes custom fields. | No for first mobile release. | Phase 2 foundation implementation. | Keep values scoped by definition and entity id; avoid duplicating canonical columns as custom values. |
| `Activity` | Shared timeline event for operational work across CRM, support, sales, purchasing, HR, and integration workflows. | CRM `Interaction`, `EventLog`, dispatch audits, webhook deliveries, business owner override audit, and loyalty transactions overlap with activity-style history. | Structured shared activity entity for cross-module operational timeline; existing domain-specific ledgers/audits remain authoritative. | Customer, Business, Order, Invoice, Supplier later; support/admin views first. | Medium if exposed in business/mobile support flows; otherwise low. | No for first mobile release unless mobile support timeline becomes launch scope. | Phase 2 foundation implementation. | Do not replace loyalty transactions, invoice archives, provider operation records, or compliance audit records with generic activities. |
| `Note` | Store human-entered notes consistently with visibility, sensitivity, and ownership. | Notes exist on Customer, Lead, Opportunity, Order, TaxCategory, LoyaltyPointsTransaction, BusinessLocation, BusinessInvitation, Supplier, and other records. | Either structured note entity for multi-note timelines or explicit note columns for simple single-note fields; use real columns where existing forms need quick display/filtering. | Customer, Business, Order, Invoice, Supplier, HR later. | Medium if notes are exposed to member/business mobile contexts. | Yes only where current mobile-facing notes are already exposed and need sensitivity review; broad note model can defer. | Phase 1 sensitivity review for mobile-exposed notes; Phase 2 shared note implementation. | Default: operator/internal notes should not be shown to mobile unless the contract explicitly states a member-safe note. |
| `Attachment` / `DocumentRecord` | Track business documents, files, generated artifacts, and uploaded evidence without duplicating media/archive concepts. | `MediaAsset`, invoice archive/source-model artifacts, shipment label storage, analytics export files, and object-storage services cover specialized document cases. | Structured document metadata entity plus object-storage key/reference; keep specialized archive/label/export records where they carry compliance or provider lifecycle. | Invoice artifacts, order documents, business media/evidence, customer documents, HR later. | Medium for invoice/mobile downloads and business media. | Yes only for invoice/order artifact compatibility; broad document model can defer. | Phase 1 compatibility decision for member artifacts; Phase 2 shared document model. | Do not move compliance archive ownership without preserving retention, hash, source-model, and download contracts. |
| `NumberSequence` | Generate consistent business numbers for orders, invoices, purchase orders, shipments, stock documents, journal entries, and HR records. | `Order.OrderNumber` exists; invoice and finance numbering needs broader policy before ERP expansion. | Structured entity with scope, document type, prefix pattern, next value, reset policy, locking/concurrency rule, and preview format. | Order, Invoice, Sales/Purchasing documents, Inventory movements, Finance documents. | Medium if member-visible order/invoice numbers change. | Yes only if current order/invoice number formats must change before mobile release; otherwise defer. | Phase 2 foundation implementation before sales/finance expansion. | Existing member-visible numbers must remain stable after issue; new sequence policy should be additive for future documents. |
| `BusinessEvent` / `AuditTrail` | Provide audit-safe event history for important state changes and future automation/AI context. | `EventLog`, provider operation records, webhook delivery, dispatch audit, business owner override audit, loyalty transactions, invoice archive metadata, and row-version coverage provide partial audit evidence. | Structured append-only business event for cross-module facts plus specialized audit records where compliance or provider behavior requires richer shape. | User/account lifecycle, Business status, Customer bridge, Order/Invoice state, Loyalty scan outcomes, Payment/Shipment events. | High if used to prove release-sensitive mobile/loyalty state transitions. | Yes for documenting required audit coverage decisions; implementation only where current release-sensitive evidence is insufficient. | Phase 1 audit gap decision; Phase 2 shared event implementation. | AI and automation may read scoped events later, but normal application commands remain the only mutation path. |
| `FeatureArea` / module visibility | Support logical module separation through UI, permissions, and feature visibility without splitting projects or databases. | Permissions, roles, WebAdmin navigation, mobile launch guards, and business access-state gates already implement parts of visibility and access control. | Structured feature-area registry or configuration-backed catalog with permission mapping, UI visibility, module flags, and default enablement. | WebAdmin navigation, business/member UI visibility, ERP modules later. | Medium if mobile hides or shows release-sensitive workflows. | Yes only for mobile/business launch gates; broad ERP module visibility can defer. | Phase 1 for release gates if gaps exist; Phase 2 for ERP module rollout. | Default deployment assumes most customers use the full system; module separation remains logical through UI and permissions. |

## Release-Sensitive Foundation Decisions

| Surface | Decision | Before mobile release action | Implementation priority | Notes |
| --- | --- | --- | --- | --- |
| Address foundation | Canonical address design is required before release-sensitive implementation. | Define one canonical structured address shape and mapping rules for `Address`, `CustomerAddress`, `BusinessLocation`, and immutable order/invoice snapshots. | P0 | Existing snapshots remain immutable; compatibility mapping must protect mobile profile, discovery, order, and invoice views. |
| Mobile API boundary | Current routes and DTOs are compatibility-sensitive. | Any breaking route or contract change must be accepted only with contract serialization tests, source-route tests, and a compatibility/versioning decision. | P0 | ERP/back-office routes must be additive and audience-scoped. |
| Loyalty ledger and scan compatibility | Loyalty ledger remains the points source of truth. | Validate QR/session, accrual, redemption, reward, campaign, and mobile summary contracts; implement required changes before release or record explicit no-change evidence. | P0 | CRM/Customer must not own a second points balance. |
| Identity profile | `User` owns linked member identity/profile/contact fields. | Decide whether any identity/profile fields need schema or contract changes before release; otherwise freeze current mobile profile contract. | P0 | External/provider tokens and secrets stay outside domain records and logs. |
| Business and membership | `Business` remains merchant root; `BusinessMember` remains mobile access source. | Decide future `Employee` linkage without replacing current business access contracts. | P0 | HR records can link later, but should not become mobile access source in the first implementation. |
| Customer bridge | Linked customers resolve profile/contact data from `User`; unlinked customers keep CRM fallback data. | Document and implement only required source-of-truth fixes before release. | P0 | Customer may support tax, segmentation, and CRM lifecycle; it does not own loyalty balances. |
| Order and invoice snapshots | Existing order/invoice models evolve; no parallel sales invoice model. | Preserve member order/invoice contracts, archive/source-model download behavior, and immutable snapshots. | P0 | Later sales and finance expansion should extend current records and projections. |

## Implementation Phase Defaults

| Phase | Meaning | Allowed changes |
| --- | --- | --- |
| Phase 1 release-sensitive foundation | Changes required before mobile release to avoid later downtime, breaking contracts, disruptive migrations, or duplicated source-of-truth. | Address canonicalization, mobile API compatibility fixes, loyalty scan/ledger compatibility fixes, identity/customer/business bridge fixes, and order/invoice snapshot compatibility fixes. |
| Phase 2 ERP foundation primitives | Shared primitives needed before broad ERP modules are implemented. | `ExternalSystem`, broad `ExternalReference`, `SyncState`, `SyncConflict`, custom fields, shared notes/activities, document records, number sequences, business events, and feature-area visibility. |
| Phase 3 module expansion support | Module-specific usage after foundation primitives exist. | CRM expansion, sales documents, purchasing lifecycle, inventory ledger/tasks, finance/accounting, HR/time, and AI-readiness workflows. |
