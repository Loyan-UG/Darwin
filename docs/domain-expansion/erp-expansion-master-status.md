# ERP Expansion Master Status

## Summary

This document is the compact decision status for the ERP redesign and expansion track. It is a planning and reconciliation document, not a replacement for the historical implementation ledger, module audit, or code-backed go-live status.

Darwin remains a single coherent ERP product. Module separation stays logical through UI navigation, permissions, feature visibility, ownership rules, and internal services. No project split or database split is introduced only to model modules.

## Active Next Gate

`Production Go-Live Evidence Execution`

The next ERP expansion step is production go-live evidence execution, starting with a production-like staging rehearsal before real production. The current decisions select MinIO as the first production object-storage target, Azure Blob readiness as the next storage hardening target after MinIO, dual-format e-invoice readiness for both ZUGFeRD/Factur-X and XRechnung, Android-first mobile launch, no real AI provider for now, finance export file-delivery as the current accounting delivery path, and implementation of provider-neutral `SyncState`/`SyncConflict` foundation. Target-specific inbound sync, accounting API adapters, real AI providers, and direct operational AI execution still require concrete targets or command families. The accepted direction remains conservative and auditable:

- classify sensitive fields and deny raw sensitive HR, payroll, bank, provider, credential, and private document data by default;
- expose only scoped, purpose-bound, privacy-safe data views to AI workflows;
- store recommendations and action drafts as evidence, not as direct mutations;
- require human approval before normal application command handlers execute operational changes;
- keep existing command handlers as mutation owners for Sales, Finance, Purchasing, Inventory, HR, Payroll, Treasury, and WebAdmin workflows;
- keep provider credentials, tokens, raw payloads, document contents, bank details, payroll internals, and private HR material out of prompts, logs, metadata, events, tests, and documentation;
- keep public/mobile/storefront, finance export package format, invoice archive/download, payment/refund, supplier finance, payroll provider submission, and WebAdmin operational flows unchanged unless a later implementation slice explicitly changes them with tests.
- keep the provider foundation internal and provider-neutral, with test-scoped deterministic adapters only and no real provider credentials, raw prompt/completion persistence, WebAdmin credential UI, public/mobile route, or autonomous execution.
- keep approved-draft handoff infrastructure limited to internal timeline evidence, internal follow-up task creation, and internal module review task routing until a direct operational executor is separately designed and approved.
- use the implemented SyncState/SyncConflict foundation for future inbound or two-way targets without adding a target-specific sync adapter prematurely;
- keep accounting API adapters blocked until concrete accounting targets exist.

## Operating Rules

- Update `BACKLOG.md` before starting a major ERP slice when the backlog no longer matches the current implementation state.
- Update the relevant domain-expansion design document before schema, migration, public contract, mobile contract, or WebAdmin mutation changes.
- Keep public WebApi, mobile/member, storefront, invoice archive/download, and issued snapshot compatibility guarded for every Sales, Finance, Purchasing, and Inventory change.
- Use real columns for common, reportable, compliance-relevant, accounting-relevant, inventory-relevant, integration-key, or cross-module fields.
- Use custom fields or JSON for customer-specific, uncertain, low-frequency, provider-specific payload, industry-specific, or unstructured metadata.
- Never store provider credentials, access tokens, refresh tokens, private keys, connection strings, or raw sensitive payloads in domain metadata, document metadata, external references, logs, tests, or documentation.
