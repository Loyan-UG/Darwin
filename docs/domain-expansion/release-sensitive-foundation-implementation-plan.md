# Release-Sensitive Foundation Implementation Plan

This document converts the foundation primitive design into the P0 implementation plan that must be resolved before mobile release. It is a documentation-only planning artifact: it does not change entities, migrations, API routes, DTOs, mobile contracts, or test code.

If a required loyalty, mobile, or foundation change is discovered during the implementation slices below, it must be completed before mobile release. Delaying mobile release is acceptable when the alternative is post-release downtime, breaking mobile contracts, disruptive data migration, or duplicated source-of-truth state.

## Current Implementation Baseline

- Address-like data currently exists in `Address`, `CustomerAddress`, `BusinessLocation`, and order/invoice snapshot payloads. These shapes must be mapped deliberately before broader ERP records depend on them.
- `User` owns identity, member profile, contact, verification, preference, token, external-login, and device-facing flows for member clients.
- `Business` is the merchant root. `BusinessMember` is currently the main source for business mobile access and must remain the release-time access source.
- `Customer.UserId` already bridges CRM customers to identity users, while fallback CRM profile and address data still exist for guest, lead, and unlinked customer workflows.
- The loyalty ledger, scan sessions, QR/session records, and loyalty mobile contracts are release-sensitive. Loyalty points remain ledger-owned.
- `ApiRoutes`, mobile shared services, launch guards, and serialization/source-contract tests define the mobile compatibility boundary.
- `Order` and `Invoice` already carry member-facing projections and immutable snapshot/archive behavior. Later sales and finance work must evolve these records rather than create a parallel invoice or order model.

## Implementation Rules For The Next Slice

- Start with `Address Foundation Compatibility Slice` before ERP sales, purchasing, inventory, finance, or HR expansion.
- Any P0 code change must include its matching tests in the same implementation step.
- If a mobile-facing contract must change, update the shared contract type, WebApi mapper/controller, mobile shared service, launch guard or source-contract test, and serialization compatibility test together.
- Existing issued order/invoice snapshots and archive payloads are historical evidence and must remain readable after mapping changes.
- JSON/custom metadata is allowed only for uncertain, customer-specific, provider-payload, or unstructured data. Common, reportable, filterable, compliance-relevant, integration-key, or cross-module fields should be structured columns/entities.
- Module separation remains logical through UI, permissions, and feature visibility. Do not split projects or databases for this release-sensitive foundation work.

## Next Implementation Slice

`Address Foundation Compatibility Slice` is the first real implementation slice.

Goal: design and implement canonical address mapping and compatibility tests for `Address`, `CustomerAddress`, `BusinessLocation`, member address contracts, business location contracts, and order/invoice address snapshots before any ERP sales, purchasing, inventory, or finance expansion.

Expected output of that slice:

- One canonical address shape and mapping rule documented in the domain design.
- Compatibility-preserving changes for member profile addresses and business locations.
- Explicit snapshot handling for order/invoice address payloads.
- Focused tests proving mobile profile, business discovery, and member commerce address behavior remain stable.
